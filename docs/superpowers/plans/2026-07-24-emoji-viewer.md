# Plan: color emoji rendering in the Viewer (app-side, no OS mods)

Renders color emoji inline in Viewer note bodies from a bundled 16x16 glyph
pack (already generated: `res/emoji/p0..p123.png` strips + `index.bin`,
3,944 RGI glyphs, ~460 KB). Also hardens UTF-8 note encode/decode against
Symbian CESU-8 so astral emoji survive the byte<->String boundary at all.
Design was fixed by the 2026-07-24 deep-research pass; do not re-litigate.

## Global Constraints

- **Language**: Java 1.3 source / CLDC 1.1 API only. NO generics, for-each,
  autoboxing, varargs, enums, annotations, StringBuilder (use StringBuffer),
  java.util collections beyond Vector/Stack/Hashtable/Enumeration,
  String.split/replaceAll/regex, java.text, exception chaining,
  Character.isWhitespace, Character.codePointAt (does not exist).
- **Source files ASCII-only** — emoji in tests/comments must use `\uXXXX`
  escapes, never literal UTF-8.
- Every method under ~150 lines (device verifier limit).
- `nok.core.*` must not import `javax.*` (enforced by test.sh compiling core
  against the desktop JDK). `getResourceAsStream` via `Class` is allowed.
- Build: `./build.sh` (JDK8 at tools/jdk8, -source/-target 1.3, ProGuard
  -microedition preverify). Tests: `./test.sh` — desktop JDK8 run of
  test/*.java suites; each suite is a `main()` that throws RuntimeException
  on failure and prints `ALL PASS <n>`; suites are listed explicitly in
  test.sh's for-loop.
- Match surrounding code style: flat classes, public fields on records,
  dense explanatory comments stating constraints (not narration), Theme.*
  colors read at layout/paint time.
- Glyph pack contract (already on disk, generated by tools/gen-emoji.py —
  read its docstring for the authoritative index.bin layout): pages are
  32-glyph 512x16 PNG strips `p<N>.png`; glyph id g lives on page g/32 slot
  g%32 at x=(g%32)*16; index.bin is big-endian with magic "NKEM", header
  (version, glyphPx=16, perPage=32, maxUnits, pageCount, singleCount,
  seqCount, glyphCount), singles table of {u16 unit, u16 glyphId} sorted by
  unit, sequence offset table (u32 x seqCount+1, in code units), glyphId
  table (u16 x seqCount), and a UTF-16 code-unit blob of keys sorted
  lexicographically by unit values. Keys are RAW code-unit sequences
  (fully-qualified AND minimally-/un-qualified variants both present, both
  mapping to the same glyph) — the runtime does greedy longest RAW match,
  no FE0F normalization pass.

## Task 1 — nok.core.Utf8: hand-rolled UTF-8 codec + wiring

**Problem**: note content crosses byte<->String via the platform codec
(`new String(b, "UTF-8")` / `getBytes("UTF-8")`). Path.java:177 documents
that some Symbian CLDC VMs emit CESU-8 (surrogate halves encoded as two
3-byte sequences) instead of proper 4-byte UTF-8; the decode side is equally
untrusted. Astral emoji must survive both directions deterministically.

**Create `src/nok/core/Utf8.java`** (no javax imports):
- `public static String decode(byte[] b, int off, int len)`:
  - 1/2/3/4-byte UTF-8 forms; 4-byte forms above U+FFFF emit surrogate pairs
    (manual arithmetic, no codePointAt).
  - CESU-8 tolerance: a 3-byte form decoding into U+D800..DFFF is emitted
    as that char unchanged — a CESU-8 pair of such forms therefore
    reconstitutes a valid surrogate pair. Do NOT reject them.
  - Invalid bytes (bad continuation, truncated tail, overlong forms of
    ASCII, lone 0x80-0xBF, 0xF8+ lead): emit U+FFFD, resync at next lead
    byte. Never throw.
  - Convenience overload `decode(byte[] b)` = whole array.
- `public static byte[] encode(String s)`:
  - Proper UTF-8: surrogate PAIRS become one 4-byte sequence (mirror the
    arithmetic in Path.segEnc, Path.java ~line 177); unpaired surrogate
    halves encode as U+FFFD's bytes (EF BF BD).
  - Byte-identical to desktop `String.getBytes("UTF-8")` for any well-formed
    input (test asserts this against the real JDK on a corpus).
- Keep both methods allocation-lean: one sizing pass or StringBuffer/byte
  scratch growth, no per-char arrays.

**Wire it in** (replace platform codec on all note-content and crypto-input
paths):
- `src/nok/NoksidianMIDlet.java:755` and the `new String(b)` fallback at
  :759 -> `Utf8.decode(b)` (the try/catch around UnsupportedEncodingException
  becomes unnecessary); `:942` readText -> `Utf8.decode`; `:960` writeText ->
  `Utf8.encode(text)`.
- `src/nok/sync/Sync.java:1232` `utf8b()` body -> `return Utf8.encode(s);`.
- `src/nok/core/VaultCrypto.java:363` `utf8()` body -> `return
  Utf8.encode(s);` (keeps device-side path-key derivation byte-identical to
  the desktop tools/nokcrypt.py reference for emoji/accented names).
- Then grep the whole of src/ for remaining `"UTF-8"` codec uses and route
  any note-content/crypto ones through Utf8; leave Path.segEnc's private
  encoder alone (it percent-encodes inline; a comment cross-referencing
  Utf8 is enough).

**Tests — create `test/TestUtf8.java`, add `TestUtf8` to test.sh's suite
list** (TDD: write tests first, watch them fail to compile/run):
- Round-trips: ASCII, Slovenian `čšž`, U+1F600 (`😀`),
  U+10FFFF, mixed text.
- encode() byte-equality vs `s.getBytes("UTF-8")` on the desktop JDK across
  a corpus of well-formed strings (loop, compare arrays).
- decode() equality vs `new String(b, "UTF-8")` for well-formed proper
  UTF-8 corpus.
- CESU-8: bytes ED A0 BD ED B8 80 (CESU-8 of U+1F600) decode to
  `😀`; re-encode yields proper F0 9F 98 80.
- Invalid inputs -> U+FFFD, correct resync, never throws: truncated 4-byte
  lead at end; lone continuation; overlong `C0 80`; `F8` lead.
- encode of a lone high surrogate -> EF BF BD.
- Suite prints `ALL PASS <n>`.

**Also**: add a `## nok.core.Utf8` section to CONTRACTS.md (mirror the
style of neighboring sections; document the CESU-8-tolerant decode and the
"byte-identical to J2SE for well-formed input" guarantee).

## Task 2 — nok.core.Emoji: index loader + greedy matcher

**Create `src/nok/core/Emoji.java`** (no javax imports; loads
`/emoji/index.bin` via `Emoji.class.getResourceAsStream` + DataInputStream,
lazily on first use, synchronized; a missing resource leaves the class in a
permanent "no emoji" mode where match() always returns 0 — never throws).

Public API (exact signatures):
- `public static boolean maybe(String s, int i)` — cheap gate before
  match(): true iff `s.charAt(i) >= 0x2000` OR (charAt(i) is `#`, `*` or
  `0`-`9` AND i+1 < length AND charAt(i+1) is U+FE0F or U+20E3). No index
  load, no allocation.
- `public static int match(String s, int i)` — packed result:
  `0` = no emoji here (draw text as today);
  else `(unitsConsumed << 16) | glyphField` where glyphField is a glyph id
  (0..glyphCount-1) or `Emoji.INVISIBLE` (0xFFFF = consume silently, draw
  nothing).
  - Greedy longest raw match: for len from min(maxUnits, remaining) down
    to 2, binary-search the sequence blob for the exact key s[i..i+len);
    on hit return (len<<16)|gid. Then the singles table for s.charAt(i).
    Compare keys char-by-char against the blob with no substring
    allocation.
  - If nothing matched, classify for the INVISIBLE/0 decision, replicating
    today's Ui.isUndrawable stripping semantics exactly (Ui.java:305):
    - U+200D, U+FE0E, U+FE0F, U+20E3 -> INVISIBLE, 1 unit.
    - a well-formed surrogate pair -> INVISIBLE, 2 units (unknown astral
      is stripped today); a LONE/malformed surrogate half -> INVISIBLE, 1.
    - BMP chars in Ui.isUndrawable's ranges (2600-27BF, 2300-23FF,
      2B00-2BFF, 2122, 2139 — read Ui.java to copy the exact set) ->
      INVISIBLE, 1 unit.
    - everything else -> 0. (U+2013 en-dash, U+2026 etc. must return 0 —
      they draw fine on the E71.)
- `public static final int INVISIBLE = 0xFFFF;`
- Geometry accessors for the Viewer: `glyphPx()`, `perPage()`,
  `pageCount()`, `glyphCount()`, and `pageOf(int glyph)` / `slotOf(int
  glyph)`.
- Resident index memory stays primitive arrays (char[] blob, int[] offsets,
  short/char[] gids, char[] singles) — roughly 75 KB; no Hashtable of
  Strings, no per-call allocation in match().

**Tests — create `test/TestEmoji.java`, add to test.sh AND change test.sh's
run classpath from `-cp build/test` to `-cp build/test:res` so
`/emoji/index.bin` resolves during tests** (uses the real committed index):
- `😀` (U+1F600) -> len 2, valid glyph id < glyphCount.
- `☀` -> len 1 hit; `☀️` -> len 2 hit; both return the SAME
  glyph id (alias contract).
- Keycap: `#️⃣` -> len 3; `#⃣` -> len 2; same glyph.
  Bare `#` and bare `5` -> 0. `maybe("#x",0)` false, `maybe("#⃣",0)`
  true, `maybe("a",0)` false.
- Flag SI `🇸🇮` -> len 4.
- Skin tone: `👍🏽` (thumbs up + medium) -> len 4, and
  its glyph differs from bare `👍`'s.
- RGI ZWJ family `👨‍👩‍👧` ->
  one match consuming all 8 units.
- Non-RGI ZWJ (`😀‍😀`): first match len 2 =
  grinning glyph; match at the ZWJ -> INVISIBLE len 1; then grinning again
  (the sanctioned component fallback).
- Lone `️` -> INVISIBLE 1; lone `\uD83D` -> INVISIBLE 1;
  `–` -> 0; plain ASCII -> 0.
- Glyph geometry: pageOf/slotOf consistency for glyph 0, 31, 32, last.
- Suite prints `ALL PASS <n>`.

**Also**: `## nok.core.Emoji` section in CONTRACTS.md documenting the API,
the packed-int contract, INVISIBLE semantics and the index layout pointer

## Task 3 — Viewer integration: K_EMOJI draw items + page cache

All in `src/nok/ui/Viewer.java` unless noted. Read the flow-state comment
block (~line 193) and flowText/flowWord/flowFrag (~1057-1212) first.

- New draw-item kind `K_EMOJI = 7` and a new `int glyph` field on DrawItem
  (Viewer.java:2494). Emoji items carry no text/font.
- **Flow**: in `flowText` (Viewer.java:1057) — the single place span text
  becomes words — scan each space-delimited segment for emoji: at each
  position where `Emoji.maybe(s, j)` is true, call `Emoji.match(s, j)`.
  On a glyph match: flush the text run before j as a word (existing
  flowWord path, preserving the current leadingSpace/paintLeadSpace
  semantics for whatever precedes), then emit the emoji as its own
  unbreakable token via a new `flowEmoji(int glyph, ...)`; INVISIBLE
  results consume their units and emit nothing; 0 falls through (the char
  stays in the text run). Words never contain emoji afterward, so
  flowWord/flowBroken stay emoji-free.
- **flowEmoji**: box width `EMW = 16*factor + 2` (1 device px breathing
  room each side; glyph drawn at x+1), height `16*factor`; wraps to a new
  line exactly like a word that doesn't fit (reuse the flowWord wrap
  logic: leading space width if a space preceded it, flowNewline when
  x + spW + EMW > flRight); sets flH to at least 16*factor; registers a
  link hit box when inside a link span (same as flowFrag does).
  DrawItem.h = max(font height, 16) * factor is NOT wanted — keep
  it.h = 16*factor and it.y = flTop; the E71 body font is 15-19 px so the
  optical mismatch is nil, and line height only grows via flH.
- **Paint** (`drawItem`, Viewer.java:1701): `K_EMOJI` branch fetches the
  page image and `g.drawRegion(page, slot*16, 0, 16, 16, 0 /*TRANS_NONE*/,
  it.x + 1, y + (it.h - 16*factor)/2 ... )` — at factor==1 draw the region
  directly; at factor>1 draw a cached nearest-neighbor upscale (see below).
  A page that failed to load draws nothing (blank 16px box, no crash).
- **Page cache**: `Hashtable` pageNo->Image plus a Vector for LRU order,
  capacity 8 pages (256 KB decoded). Load via
  `Image.createImage("/emoji/p" + n + ".png")` in try/catch (IOException
  AND OutOfMemoryError: on OOM clear the whole cache and retry once, then
  give up for that paint pass). Evict least-recently-USED (touch on every
  draw). Cache is static-per-Viewer-instance (field), NOT cleared by
  setNote — pages are note-independent.
- **factor > 1**: upscale the 16x16 region nearest-neighbor to
  16*factor, cached in the existing `glyphCache` under key
  `"E|" + glyph + "|" + factor` (check how glyphCache is keyed/cleared —
  scaledRun at Viewer.java:1816 — and follow the same lifecycle; note
  glyph pixels must come from the page via getRGB into an int[], scaled
  with the same integer-block technique as scaledRun's upscale; alpha is
  preserved by createRGBImage(..., true) — the emulator dims such images
  ~0.75x via its LCD filter, which is a known emulator-only artifact, do
  not "fix" it).
- **Unchanged on purpose**: Ui.plain/Ui.clip callers (Library, lists,
  titles, table cells, link-label chips at Viewer.java:737) keep stripping;
  fenced CODE blocks keep drawing raw text. Emoji render only in flowed
  spans (paragraphs, headings, quotes, callouts, list items, inline code
  chips — anything that reaches flowText).
- **build.sh**: in the packaging step after the icon cp, add
  `cp -r res/emoji build/pre/emoji` (JAR entries `emoji/p*.png`,
  `emoji/index.bin`). Run `./build.sh` and report the new JAR size (expect
  ~660-700 KB, budget hard cap 1 MB).
- **Compile-check discipline**: `./test.sh` still passes (core untouched by
  this task except none), `./build.sh` compiles Viewer against the CLDC
  bootclasspath — this is the only compile gate for ui code; there are no
  UI unit tests. Manual-test note: not required in this task; the
  controller runs the emulator smoke test after review.
- **CONTRACTS-UI.md**: document K_EMOJI, the page-cache bound (8 pages),
  the flowEmoji wrap semantics, and the "lists/tables still strip" rule in
  the Viewer section.
